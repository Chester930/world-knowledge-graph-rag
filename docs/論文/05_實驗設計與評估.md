## 5.3 前置參數校準實驗：Chunk Size 敏感度

> 狀態：🟡 草稿（2026-07-17 新增）。此實驗**不對應任何正式 RQ**——chunk size 是工程超參數，不是研究問題本身——但它是 5.2 節所有消融組別共用的前提：3.1.1 `CMP` 分類分數、3.1.1 §a HDBSCAN 分群、3.2 §a 路由層向量粗篩、以及最終 RAG 問答品質，全部建立在「文字先被切成 chunk」這個步驟之上。目前的切塊參數（`parser/core.py` `sentence_aware_chunking(chunk_size=500, chunk_overlap=50)`）是沿用 v1 的經驗值，從未做過系統性驗證——若直接拿這個未驗證的參數去跑 RQ1-RQ6 的正式消融實驗，任何負面結果都無法排除「是不是切塊切壞了」這個混淆變因。因此本實驗**必須排在 5.2 節正式消融之前執行**，其結論（鎖定的 chunk size 數值）是後續所有實驗的固定前提，不是與 RQ1-RQ6 平行的另一組消融。

### 5.3.1 文獻依據與一個必須誠實處理的落差

- **Chen et al.（2023/2024，EMNLP 2024）Dense X Retrieval**：細粒度檢索單位（proposition 級）在檢索準確率與下游問答表現上系統性優於粗粒度段落，支持「切更細」的方向。
- **Bhat et al.（2025）**：切塊大小的最佳值**因任務類型而異**——事實型問答適合小塊（64-128 tokens），需要廣泛脈絡理解的任務適合大塊（512-1024 tokens）；且不同 embedding 模型對切塊粒度的敏感度不同。

> ⚠️ **單位落差，不可直接套用**：上述兩篇文獻回報的切塊大小皆以英文 **token**（子詞切分）為單位，但 `sentence_aware_chunking()` 目前的 `chunk_size` 參數計的是 Python **字元數**（`len(part)`），而本專案語料以中文為主。中文字元與子詞 token 的對應比例並非 1:1（且因分詞器而異），直接把「64-128 tokens」代入當作「64-128 個中文字元」是錯誤的單位轉換，會讓實驗設計本身站不住腳。本實驗的變因因此設計為**字元數的掃描區間**，並在每組設定實際跑一次 `LocalEmbeddingProvider` 底層 `sentence-transformers` 分詞器，回報測得的真實 token 數，讓「字元數 ↔ token 數」的對應關係是**實測值，而非假設值**，寫作時才能誠實地與文獻的 token 區間做對照，而不是含糊帶過單位問題。

### 5.3.2 實驗設計

**自變因**：`chunk_size`（字元數），掃描以下 5 組設定（涵蓋現行預設與文獻建議方向的粗略對應區間，兩端皆比現行預設更細/更粗）：

| 組別 | `chunk_size`（字元） | `chunk_overlap`（字元） | 備註 |
|---|---|---|---|
| S1 | 150 | 50（固定） | 對應文獻「小塊、事實型任務」方向 |
| S2 | 300 | 50（固定） | |
| S3（現行預設） | 500 | 50（固定） | v1 經驗值，作為對照基準 |
| S4 | 800 | 50（固定） | |
| S5 | 1200 | 50（固定） | 對應文獻「大塊、脈絡理解任務」方向 |

**控制變因**（單一變因原則，比照 3.6 節）：
- `chunk_overlap` 固定為現行預設值 50 字元，不在本次消融範圍內——若五組主結果顯示 chunk size 有顯著影響，overlap 的獨立消融可列為後續延伸，避免本次實驗變成二維網格搜尋、稀釋每組樣本的統計檢定力。
- **Embedding provider／model 全程固定**為系統預設（`local`，`core/embedding_guard.py` 已在啟動時強制單一 provider，見第三章 3.1 節），不在本次實驗內切換 provider——Bhat et al.（2025）指出不同模型對切塊粒度的敏感度不同，這代表「chunk size × embedding model」存在交互作用，但窮舉多個 provider 會使實驗矩陣倍增，超出本論文核心 RQ（RQ1-RQ3）的工作量負擔；此交互作用效應**明確列為未探索的範圍限制**，記入第七章未來工作，不可略過不提。
- 其餘系統元件（雙層路由、受控關係抽取、自我精煉迴圈）維持 Full System 預設設定不變，只有輸入語料的切塊方式隨組別改變。

### 5.3.5 分類門檻校準（`CLASSIFY_AUTO_THRESHOLD`／`CLASSIFY_MIN_THRESHOLD`）

> 與 5.3.1-5.3.4 的關係：chunk size 決定「CMP 分數本身算得準不準」，本節校準的是「同一個 CMP 分數，切在哪個數值算自動歸檔、哪個算完全無關」的**決策門檻**——概念上是獨立的問題，但依賴同一份驗證集與 5.3.1-5.3.4 已鎖定的 chunk size，故排在 5.3 節之內、5.4 節正式資料集建置之前，作為前置校準實驗的第二部分，不另立新章節。

**現況問題**：`core/constants.py` 目前的 `CLASSIFY_AUTO_THRESHOLD = 0.30` 與 `CLASSIFY_MIN_THRESHOLD = 0.05` 是針對 v1 舊公式（`cosine × alignment × magnitude`）校準的遺留值。第三章已將分類分數公式改為 Prototypical Networks 的 centroid cosine 相似度（拿掉了經查證恆為常數、無實際作用的 alignment/magnitude 加權，見第三章 3.1.1 節），數值尺度不保證與舊公式相同——這兩個數字目前純粹是沿用舊值的佔位符，**不是通過任何驗證的結果**（程式碼中已加上對應警告註解，見 `core/constants.py`）。

**為什麼文獻無法回答這個問題**：本論文引用的 Prototypical Networks（Snell et al., 2017）與 Sentence-BERT（Reimers & Gurevych, 2019）驗證的是「用 cosine 相似度衡量語意距離」這個**計算方式**本身有效，但沒有任何文獻能告訴你「你的語料、你選定的 embedding 模型算出來的分數，切在 0.3 還是 0.4 才是對的」——**這是一個資料集依賴的數值，只能靠實際跑一次帶標籤的驗證資料才能定案，查文獻查不到答案**。這點必須在論文中誠實聲明，不能用「有文獻佐證分類機制」模糊帶過「門檻數值本身未經驗證」這件事——機制有效不代表你設的切點有效，這是兩個不同層次的問題。

**實驗設計**：
1. 沿用 5.3.3 節「分類準確率」所需的同一份驗證集（一批已知正確歸屬 KG 的文件），並擴充至少一批**刻意不屬於任何現有 KG 的負樣本**，用來檢驗 `CLASSIFY_MIN_THRESHOLD` 能否正確排除無關文件。
2. 固定 5.3.1-5.3.4 節已鎖定的 chunk size 與系統預設 embedding provider，只掃描門檻值本身：對每份驗證文件計算其與各 KG 的 CMP 分數，畫出分數分佈（正確歸屬 vs. 錯誤歸屬 vs. 完全無關三組），觀察三組分數是否有清楚可分的區間。
3. 以類似 PR 曲線的方式，掃描候選 `CLASSIFY_AUTO_THRESHOLD` 值（例如 0.10 至 0.60，間隔 0.05），計算對應的自動分類 precision（自動歸檔的文件裡真正歸對的比例）與 recall（本應自動分類、卻因門檻太高被誤放進 SUGGEST/POOL 的比例），取兩者平衡點作為正式值；`CLASSIFY_MIN_THRESHOLD` 則以「無關負樣本分數的上界」為依據校準，確保明顯無關的文件不會被誤判進 SUGGEST 清單。

**既有設計已吸收部分門檻誤差風險，這是不採用自適應機制的額外理由**：第三章 3.1.1 節的三層信心分級（AUTO／SUGGEST／POOL）本身就是針對「門檻不可能完全準確」這個事實的緩解機制——分數落在兩個門檻之間的文件不會被錯誤地自動歸檔，而是進入 SUGGEST 清單交由使用者確認再拍板。這代表門檻校準**不需要追求完美的單一切點**，只需要讓 AUTO 區間的 precision 夠高（避免誤自動歸檔）、MIN 區間夠保守（避免真正相關的文件被過早放棄留在暫存區），中間的灰色地帶本來就設計成由人工把關。也因此，門檻校準採「一次性驗證＋鎖定數值」而非「線上自適應調整」是合理選擇——現有設計已有一層人工審核吸收門檻誤差的代價，疊加一層機器學習機制去動態調整門檻，只會讓「這個門檻為什麼是這個數字」變得更難向讀者交代，卻換不回相應的必要性（同樣的理由已在 `core/providers/embedding/README.md` 討論過模型一致性與 chunk size 時說明過，此處是同一原則的第二次應用）。

**產出**：校準後的兩個門檻值連同其 precision/recall 曲線數據，一併鎖入 `core/constants.py`（取代現有帶警告註解的佔位值）與第五章實驗結果附錄，作為後續 RQ1-RQ6 消融實驗中涉及 3.1.1 分類正確性部分的固定前提。`KG_ROUTE_THRESHOLD`（3.2 §a 路由層門檻，對應 RQ2）是同一類型的舊公式遺留值，但因其直接對應正式 RQ2，校準實驗歸入 5.2 節 RQ2 消融組別內處理，不併入本節。
