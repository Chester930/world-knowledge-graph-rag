## 2.2 各研究問題（RQ1-RQ6）核心文獻總覽

本節逐條列出六個研究問題（RQ1-RQ3 確定、RQ4a/RQ4b/RQ5/RQ6 預留）各自的核心參考文獻與其扮演的角色，細節留待 2.4 節逐階段展開，完整書目與本地檔案見 `docs/論文/附錄與參考文獻.md`。

**RQ1（KG-BFS 架構相較純向量 RAG 的優勢邊界）**：核心文獻為 Han et al.（2025，🟡）《RAG vs. GraphRAG》與 Xiang et al.（2025，🟢 ICLR 2026）《When to use Graphs in RAG》，兩者皆挑戰「圖結構全面優於向量檢索」的預設，主張優勢具**任務依賴性**；Fan et al.（2026，🟡）《Do We Still Need GraphRAG?》進一步指出 Agentic 迭代式搜尋能縮小與 GraphRAG 的效能差距。本論文據此設計消融實驗，界定 KG-BFS 優勢成立的具體查詢任務類型，而非重複驗證已有共識的問題。完整回顧見 2.4.3。

**RQ3（自我精煉檢索迴圈）**：核心文獻為 Self-RAG（Asai et al., 2023，🟢 ICLR 2024）、FLARE（Jiang et al., 2023，🟢 EMNLP 2023）、IRCoT（Trivedi et al., 2022，🟢 ACL 2023）三篇主動式/反思式檢索文獻，本論文延伸其評估範疇至結構化三元組 Context，機制上與三者皆有明確差異。完整回顧見 2.4.7。

本節依第三章系統設計與方法論的章節順序，逐一補充該階段所採方法的文獻/專案佐證，讓「這個機制為什麼這樣設計」與「第三章的哪個節點」的對應關係一目瞭然。3.1.2-3.1.4（抽取任務佇列、抽取過程、抽取後續）三節本身純屬工程機制描述，其涉及的受控詞彙/別名消解理論已分別歸入 2.4.4／2.4.5，此處不重複列節。

### 2.4.1 對應 3.1.1（暫存區分類與自動分群）

第三章 3.1 節開頭已聲明：暫存區分類與抽取斷點續傳屬建圖流程的基礎工程機制，非本論文六項研究問題（RQ1-RQ6）的驗證對象。但 3.1.1 節（暫存區分類與資料夾歸檔）與 3.1.1 §a（AI 自動分群機制）的分類分數計算、分群演算法、命名輸入篩選三個子機制，皆刻意選擇以有廣泛驗證基礎的文獻/專案為主要參考起點，而非直接沿用 v1（智慧知識庫）的既有實作——這個查證過程本身具備方法論意義，值得獨立記錄。本節依 3.1.1 的流程順序（分類 → 分群）分兩部分整理。

#### 2.4.1.1 分類分數：質心/原型式路由

3.1.1 節 `CMP` 節點的分類分數計算，採 **Prototypical Networks**（Snell, Swersky, Zemel, 2017，🟢 NeurIPS 2017）的 centroid 相似度精神——將每個類別（本論文語境下為每個 KG）表示成其支持集（成員文件）表徵向量的平均（prototype），新樣本依其表徵向量與各類別 prototype 的距離分類。這是 few-shot 分類文獻中被廣泛驗證、實作簡單且不需訓練額外參數的方法，適合本論文「KG 數量與每個 KG 的文件數皆隨使用逐步增長、無法預先準備大量標註訓練資料」的場景。

**可信任專案背書（產業驗證，🟡 部分吻合）**：以 embedding 相似度做離散類別路由，並非只有本論文與 Snell et al.（2017）如此設計——**semantic-router**（Aurelio Labs，[aurelio-labs/semantic-router](https://github.com/aurelio-labs/semantic-router)，3,738★，2026-07-23 GitHub API 查證）是持續維護的生產級語意路由函式庫，驗證了「查詢向量與類別代表向量比對以決定路由」這個大方向已是業界實際部署的設計模式。但查證其 GitHub Discussion #451 維護者說明後發現，該專案的實際機制是**即時 kNN**——查詢向量與每個 route 底下每一句範例語句個別比對取 cosine 距離，再對 top-k 結果做聚合（`aggregation`，可設 mean/sum）決定路由，並非預先算好單一 centroid 向量後只比對一次。**此處需誠實區分兩個層次**：semantic-router 佐證的是「embedding 相似度可作為離散分類的生產級信號」這個較粗的設計選擇，並不直接佐證 `CMP` 節點「預先計算單一 KG prototype 向量」這個具體機制——後者仍以 Snell et al.（2017）的原型網路精神為直接依據，兩者不可混為一談。

**與本論文的關係**：3.1.1 節 `CMP` 節點的 centroid 相似度分類，其演算法依據是 Prototypical Networks；semantic-router 僅作為「同一大類設計模式已有獨立生產實例」的旁證，用以說明本論文的技術選型並非脫離業界實踐的孤立設計，但不構成對 `CMP` 具體實作機制的直接驗證。

#### 2.4.1.2 AI 自動分群：HDBSCAN + LLM 命名

3.1.1 §a 的分群演算法採 **HDBSCAN**（McInnes, Healy, Astels, 2017，🟢 *Journal of Open Source Software*），命名輸入篩選採 **Khandelwal（2025，🟡）**《Using LLM-Based Approaches to Enhance and Automate Topic Labeling》驗證表現最佳的主導子群法（詳見第三章 3.1.1 §a 節）。

**參考專案（2026-07-23 補齊，區分「直接依賴」與「架構背書」兩個不同層次）**：
1. **直接依賴套件**：`services/cluster_service.py` 直接 `import hdbscan`，實際呼叫的就是論文作者本人維護的官方參考實作 [scikit-learn-contrib/hdbscan](https://github.com/scikit-learn-contrib/hdbscan)（3,132★，2026-07-23 GitHub API 查證）——這不是「架構模式的獨立佐證」，是本論文程式碼實際執行的套件本身，證據等級最高：不只是「這個演算法理論上可行」，而是「這個演算法的官方實作可以直接跑在本論文的程式碼裡」。
2. **架構背書（產業驗證，🟡 強吻合，因來源為 arXiv 預印本尚無正式期刊/會議版本）**：**BERTopic**（Grootendorst, 2022，arXiv:2203.05794）是「embedding → HDBSCAN 分群 → LLM/統計式命名」這整條管線在業界已廣泛採用、原作者持續維護（[MaartenGr/BERTopic](https://github.com/MaartenGr/BERTopic)，7,752★，2026-07-23 GitHub API 查證）的獨立實作，其標準管線為 embedding → UMAP 降維 → HDBSCAN 分群 → 主題表示（預設 c-TF-IDF 統計式命名，另支援作者本人與社群文件證實可行的 LLM-based 命名，見 Grootendorst〈Topic Modeling with Llama 2〉部落格文章）。這比單獨引用 HDBSCAN 論文本身更有力地佐證 3.1.1 §a「HDBSCAN 分群＋LLM 命名」這整條管線的架構選擇，因為 BERTopic 是獨立於本論文、已被業界實際驗證有效的完整實作，不只是演算法層級的學術文獻。

**需誠實標註的落差**：BERTopic 標準管線在 HDBSCAN 分群前固定接一段 UMAP 降維，理由是 HDBSCAN 賴以計算群集的 mutual reachability distance 在高維空間容易受「維度詛咒」影響、密度估計失真；本論文 `services/cluster_service.py` 已於 2026-07-20 補上條件式 UMAP 降維（池規模 ≥20 才套用），此落差已處理，見第三章 3.1.1 優化建議第 4 項。

**三層信心分級機制的定位澄清**：本論文另外查證了 **Paperless-ngx**（真實、廣泛使用的開源文件管理系統，[paperless-ngx/paperless-ngx](https://github.com/paperless-ngx/paperless-ngx)，43,354★，2026-07-23 GitHub API 查證）的 ML 自動分類機制，作為 3.1.1 主圖 `AUTO`／`SUGGEST`／`POOL` 三層信心分級設計的潛在對照對象，但查證其技術文件後發現該專案的分類器（scikit-learn `MLPClassifier`）**沒有信心門檻或建議層**——直接套用單一預測結果，僅提供管理員手動預覽用的 `--suggest` dry-run 旗標，並非本論文依分數分三層、`SUGGEST` 層讓使用者逐一確認候選的設計。**此查證結果的意義是反向的**：本論文找不到現有生產系統對「三層信心分級＋使用者可控確認」這個具體設計的直接先例，這使得 3.1.1 主圖的三層機制更接近本論文刻意做出、以使用者可控性優先於全自動化的工程決策，而非業界既定做法的複製——演算法（centroid 相似度、HDBSCAN、命名輸入篩選）借鏡自成熟文獻與專案，但信心分級的介面設計是本論文自行提出，兩者性質不同，不應混為一談。

### 2.4.2 對應 3.2 §a（ConceptNode 路由層，RQ2）

**GraphRAG 核心方法**：Edge et al.（2024，🟡 arXiv 預印本，微軟研究院）提出的 GraphRAG，流程為文字切片 → LLM 抽取實體與關係 → 跨片段的同一實體/關係描述整併摘要 → 建圖 → 階層式 Leiden 社群偵測（遞迴切分至無法再分的葉節點社群）→ 逐社群生成摘要報告 → 查詢時依需求選擇 **Local Search**（以特定實體為中心的局部檢索）或 **Global Search**（跨社群摘要的 map-reduce 式問答）。此設計解決了純向量 RAG 無法回答「總結全文重點」這類全局性查詢的問題，但其架構預設是**單一大圖**，對「多個獨立知識圖譜並存」的場景著墨甚少——這正是第一章 1.1.3 節已指出的具體缺口，也是本論文 RQ2（多知識庫路由）的直接出發點。

**增量更新與可擴展性的旁支討論**：GraphRAG 官方 GitHub（microsoft/graphrag#741）本身承認新增文件需要重新跑過整個社群結構，無法增量更新。**LightRAG**（Guo et al., 2024，🟢 EMNLP 2025）改用雙層檢索（低層節點對應具體實體、高層叢集對應主題關係）取代階層社群摘要；**EraRAG**（Zhang et al., 2025，🟡）以 LSH（局部敏感雜湊）分群，只對受影響的圖區域做局部重新切分。此問題屬於**工程借鏡型**改善（見 `docs/報告/03_核心架構藍圖.md` 痛點 1 分類），業界已有成熟解法可直接參考，本論文不將其列為研究問題。

**與本論文的關係**：本論文第三章 3.2 §a 節的路由層（ConceptNode）與 Microsoft GraphRAG 的「單圖＋社群摘要」是不同的架構選擇，完整比較見 2.3 表格。RQ2 驗證的是「輕量路由層＋BFS 知識層」這個具體兩層架構在準確率、延遲、可擴展性上的效能取捨——業界對照為 **WhyHow.ai** 的 namespace 隔離機制，但未見系統性效能驗證（見 1.2 節 RQ2 範圍澄清），本論文填補的是這個驗證缺口，而非提出全新的隔離機制本身。**熱度佐證與誠實侷限（2026-07-23 補齊，完整查證見 `docs/報告/產品競品研究/00_總覽.md`）**：WhyHow.ai 目前主力產品 `whyhow-ai/knowledge-graph-studio` 929★／107 forks（MIT），較早期的純 SDK `whyhow-ai/whyhow`（namespace 隔離機制實際所在的套件）僅 128★ 且已停止更新——相較 RAGFlow（85,733★）、Neo4j LLM Knowledge Graph Builder（約 4,955★）等其他業界對照，WhyHow.ai 的規模明顯小得多，「業界對照」一詞應理解為「確有此類產品存在」而非「已獲廣泛驗證的成熟做法」，這點在寫作定稿時需要誠實反映在措辭強度上。

### 2.4.3 對應 3.2 §b（BFS 圖遍歷，RQ1）

**GraphRAG 是否真的優於一般 RAG？——一個正在進行中的學界辯論。** 近兩年（2025-2026）出現多篇論文對「圖結構是否天生有利於 RAG」提出質疑，而非預設圖結構全面優於一般向量檢索：

- **Han et al.（2025，🟡）**《RAG vs. GraphRAG: A Systematic Evaluation and Key Insights》系統性實證比較顯示，GraphRAG 相對優勢是**任務依賴**的，並非全面優於一般 RAG；該文獻同時揭露 LLM-as-Judge 評估協定存在**位置偏誤**（同樣兩個答案，放在評審 prompt 的前後順序不同，評審 LLM 的偏好會改變），直接質疑 Edge et al.（2024）原始論文所用的「comprehensiveness/diversity/empowerment」LLM 評審協定的可靠性。
- **Xiang et al.（2025，🟢 ICLR 2026）**《When to use Graphs in RAG》提出的核心發現是：「近期研究報告 GraphRAG 在許多真實任務上經常表現不如一般 RAG」，並建立 GraphRAG-Bench 基準，測試圖結構究竟在哪類任務（事實檢索／複雜推理／摘要／創意生成）才真正有幫助，而非預設圖結構全面有利。
- **Fan et al.（2026，🟡）**《Do We Still Need GraphRAG?》發現 Agentic（迭代式）搜尋能大幅縮小與 GraphRAG 的效能差距，GraphRAG 的優勢主要僅在複雜多跳推理場景才站得住腳——換言之，建圖成本只在特定查詢類型下才划算，並非普遍成立。
- 另有 **FG-RAG**（Hong et al., 2025，🟢 CIKM 2025）指出社群摘要整合的是實體的「內在」資訊而非「查詢相關」資訊，**NodeRAG**（Xu et al., 2025，🟡）指出檢索一個實體會不分青紅皂白帶入所有關聯事件（顆粒度太粗），兩者都從不同角度支持「社群摘要式檢索有結構性代價」這個判斷。

**與本論文的關係**：RQ1（KG-BFS 相較純向量 RAG 的優勢邊界）直接回應 Han et al.（2025）與 Xiang et al.（2025）「GraphRAG 優勢具任務依賴性」的核心論點——本論文不泛問「圖結構是否優於向量檢索」（此問題已有上述文獻回答，結論是任務依賴的），而是在本論文的具體架構（3.2 §b 的 2-hop BFS 圖遍歷）上，系統性界定優勢成立的查詢任務類型與邊界條件，完整架構比較見 2.3 表格。

### 2.4.4 對應 3.3（受控語意關係抽取，RQ4a）

開放資訊抽取（Open Information Extraction, OpenIE）容許任意動詞字串作為關係謂語，優點是召回率高、不受限於預先定義的關係集合，代表性基準線為 **Stanford OpenIE**（Angeli et al., 2015，🟢 ACL 2015）。但開放抽取的代價是語意不一致——**CESI**（Vashishth et al., 2018，🟢 WWW 2018）明確指出，語意相同的關係常以不同字串存入知識庫（例如「創立」「成立」「建立」被記為三個不同的關係謂語），CESI 因此提出以規範化（canonicalization）技術把這些字串收斂到同一個關係節點，這是開放抽取陣營內部嘗試解決一致性問題的代表性工作。

另一條路徑是**受控詞彙標準化**——以預先定義的本體或詞彙表限定關係類型的取值範圍。**Schema.org**（Guha et al., 2016）是網頁結構化資料領域最具影響力的受控詞彙標準，其設計精神（為每種類型預定義屬性模板、允許社群提案擴展但需經審核）是本論文第三章 3.3 節「結構驗證→語意驗證」兩步機制的類比對象。**⚠️ 需注意**：Guha et al.（2016）發表於 *Communications of the ACM* 59(2)，屬 ACM 付費資源，2026-07-20 再次查證 ResearchGate／Academia.edu 皆非作者授權的合法免費全文，本專案僅記書目資訊、仍未取得全文（見 `docs/參考文獻/03_資訊抽取與本體設計/README.md`），第三章 3.3 節對 Schema.org 精神的類比，寫作定稿前仍需透過學校圖書館取得全文逐一核實。

**RQ4a 替代查證來源（2026-07-20 新增）**：在 Guha et al. 全文取得之前，本論文另外查證並精讀了 **Vrandečić & Krötzsch（2014，🟢 *Communications of the ACM* 57(10)）**《Wikidata: A Free Collaborative Knowledgebase》作為部分替代來源——此文開放取用，描述了 Wikidata 屬性（property）**必須指定 datatype**（結構層限制）、且 schema 本身與資料一樣受社群治理（"Contributors edit the population number of Rome but also decide whether there is such a number in the first place"），社群另訂定語意限制條件（如「一個項目最多只能有一個出生地」）並以外部工具掃描違規資料（語意層檢查）。這證實了「受控詞彙的結構層與語意層可分別治理」是已在生產級系統規模化運作（截至該文寫作時已有 1,176 個屬性、4300 萬筆陳述句）的真實設計模式。**誠實的適配度聲明**：Wikidata 論文並未描述 Schema.org 式「結構驗證→語意驗證」**循序、單一審核閘門**的精確機制（其 datatype 限制與語意限制檢查是兩套獨立運作的機制，非新詞彙進入系統前的循序關卡），因此**不能宣稱兩者機制等價**，僅能作為「此類設計模式有生產級先例」的獨立佐證，與 Schema.org 本身的類比仍以 Guha et al.（2016）為準——完整適配度分析見 `docs/參考文獻/03_資訊抽取與本體設計/README.md`。

**與本論文的關係**：本論文選擇受控語意關係詞彙（`SVO_REL_TYPES`，30 類，見第三章 3.3 節），是刻意驗證「封閉式抽取」這個極端選項，而非業界更常見的開放式或半開放式抽取路線。這個選擇需要對比開放式抽取的文獻（Angeli et al., 2015 為基準線，CESI 為開放抽取陣營內部的一致性補強方案）以說明 trade-off：受控集合能保證語意一致性與可追溯性，但必然犧牲部分細緻語意的召回率，此 trade-off 正是 RQ4a（預留）要實證回答的問題。

**參考專案查證（2026-07-23 補齊，結論：查無可直接對照的參考專案）**：Wikidata 背後的實際運作軟體是 Wikibase（`wmde/*` 系列），但其核心開發主要發生在 Wikimedia 自家的 Gerrit（`gerrit.wikimedia.org`），非以 GitHub 為主要協作場所，GitHub 上的鏡像/工具倉庫（如 `Wikidata-Toolkit/Wikidata-Toolkit`，411★）星數規模與 Wikidata 實際運作規模（1,176 個屬性、4300 萬筆陳述句）不成比例，**不適合**用 GitHub star 數作為熱度佐證——Wikidata 的規模應以其自身營運數字為準，已見上方 2026-07-20 段落。更關鍵的是：本論文第三章 3.1.3 節的 `STRUCT`（結構驗證）／`SEMANTIC`（語意驗證）**循序、單一審核閘門**這個具體機制設計，查無任何開源專案實作過完全對應的流程（Wikibase 的兩層檢查是獨立運作、非循序關卡，已於上方誠實聲明）——這代表 RQ4a 這部分機制**不只是缺文獻全文核實，也缺可執行的參考程式碼**，是目前系統中「文獻依據薄弱」與「參考專案空白」同時成立的少數幾個節點之一，第三章 3.1.3 節「設計提案，尚未實作」的標註與此查證結果一致，非過度保守。

### 2.4.5 對應 3.4（實體指代與別名消解，RQ4b）

RQ4a 處理的是 SVO 三元組裡「V」（關係類型）該不該受控；RQ4b 處理的是「S／O」（實體本身）的另一半問題——同一個真實世界的實體，在文字裡可能以完全不同的字面形式出現：**代名詞**（他、它、該公司）、**全稱/簡稱/別名**（Interstate Highway 35／I-35／Interstate；Richard Stone／Stone／the defendant）。第三章 3.4 節的機制分兩層，分別解決「同一份文件內」與「跨文件」兩種不同尺度的指稱不一致問題。

**句子級抽取若不處理指代消解會漏抽**：**Text2KGBench**（Mihindukulasooriya et al., 2023，🟢 ISWC 2023）是本體論式受控三元組抽取的專門 benchmark，論文自承句子級抽取會漏抽跨句指代消解關係。**CORE-KG**（Meher, Domeniconi & Correa-Cabrera, 2025，🟡 KDD '25 Workshop）提出「指代消解前置於切塊之前」的解法，其消融研究（Meher & Domeniconi, 2025，🟡）量化證實移除指代消解模組使節點重複率上升 28.25%、雜訊節點增加 4.32%（LLaMA 3.3 70B，20 份真實法院文件）——這兩篇文獻共同支撐第三章 3.4 §a「別名消解跑在代名詞消解之前」這個順序決策的大方向，但論文任務性質是開放式抽取，與本論文受控詞彙場景不同，量化數字不可直接套用。

**跨文件增量聚類架構的文獻先例**：文件內別名登記表的動態標準名提升規則（`should_promote_by_length()`，PROMOTE 節點）查無直接文獻先例，經三次修訂查證後誠實標註為本論文自行設計的工程決策（完整查證史見 `docs/參考文獻/10_跨文件實體別名消解與增量聚類/README.md`）。但「跨文件持續擴增、標準名隨語料修正」這個架構方向有真實文獻先例：**Rao, McNamee & Dredze（2010，🟢 COLING 2010）**是串流式跨文件實體指代消解的先驅論文，明確對照傳統離線貪婪聚合式聚類與串流場景的增量更新需求；**Ji, Grishman & Dang（2011，🟢 TAC 2011）**的 TAC-KBP Entity Linking track overview 是官方、跨文件、隨語料持續擴增聚類機制（NIL clustering）的最高權威先例；**Saeedi, Peukert & Rahm（2020，🟢 ESWC 2020）**提出知識圖譜場景下的增量式多來源實體對齊（FAMER 框架），與本論文「跨文件別名圖譜隨時間持續擴增」的目標場景最為貼近。**誠實侷限**：這 3 篇皆只佐證「跨文件增量聚類」的**架構層面**，皆未描述本論文需要的「標準名選取規則」具體演算法——該規則已改採頻率優先，文獻依據交叉引用 2.4.4 已有的 Wikidata（社群共識最常用名稱）與 CESI（頻率加權質心），兩層佐證分工詳見 `docs/參考文獻/10_跨文件實體別名消解與增量聚類/README.md`。

**與本論文的關係**：第三章 3.4 §a（文件內前處理）／§b（跨文件別名消解）兩節的機制設計本身——登記表比對＋分級 LLM 仲裁的具體流程——為本論文自行設計，非直接沿用 CORE-KG 或 Text2KGBench 的演算法，兩篇文獻驗證的是「指代消解該前置於切塊之前」這個大方向，不背書本論文登記表比對機制本身（見第三章 3.8 節誠實框架）。

**參考專案查證（2026-07-23 補齊）**：CORE-KG（Meher, Domeniconi & Correa-Cabrera, 2025）與其消融研究皆為 workshop 論文，2026-07-23 查證確認**未公開釋出程式碼**（GitHub 搜尋無對應官方倉庫），只能依論文文字描述核實方法，無法對照實際執行的程式碼；Text2KGBench 搜尋到疑似對應的公開倉庫（88★），但擁有者帳號與論文作者姓名不符，**未能確認是否為官方釋出版本**，暫不引用為佐證，避免誤用非官方程式碼當作論文方法的驗證依據。§a 的 `PROMOTE` 長度優先機制與 §b 的跨文件標準名更新機制，皆已於上方與 09/10 報告誠實記錄「查無直接文獻先例」，此次查證進一步確認**也查無可執行的參考專案**——3.4 節整體是本論文機制設計自主性最高、外部佐證最薄弱的一節，這與第三章「本論文自行設計的工程決策」的分類判斷一致，非查證疏漏。3.4 §b 唯一有實際程式碼可查證的參考專案是 HippoRAG（`OSU-NLP-Group/HippoRAG`，3,882★，見 `docs/ARCHITECTURE.md`「雙層圖結構」決策與第三章 3.4 §b「記錄別名而非只是靜默合併」段落），但其驗證的是 `HAS_ENTITY` 邊本身的資料模型，不是 3.4 §a/§b 的別名比對演算法。

### 2.4.6 對應 3.5（事實時序保留策略，RQ5，2026-07-23 新增）

**問題背景**：知識圖譜中的事實會隨時間改變（例如機構搬遷、人事異動），若系統不處理事實版本，新舊事實會在圖中無關聯地並存或被靜默覆蓋，破壞多跳推理準確率與時態查詢一致性——第三章 3.5 節已確認本專案現行 MERGE 機制（`kg_id, subject, rel_type, object` 完全相等才合併）對此完全沒有偵測能力，是六個研究問題中與現行系統落差最大的一項。

**核心文獻**：**T-GRAG**（Li et al., 2025，🟢 ACM Multimedia 2025）提出時序衝突與冗餘處理框架，是本論文採用的雙時態（bi-temporal）知識圖譜方法論對照對象——舊事實不刪除、標記失效區間，新事實標記生效區間，查詢時可依時間範圍過濾。

**與 `03_核心架構藍圖.md` 原引用的落差（誠實記錄）**：`03_核心架構藍圖.md` 痛點 5（2026-07-10）最初參考的是 **Zep/Graphiti**（Rasmussen et al., 2025），但該版本查證方式僅到 WebSearch 確認存在性，未下載全文；01 緒論於 2026-07-13 正式定案 RQ5 時改採已下載全文的 T-GRAG。兩篇文獻皆屬「雙時態知識圖譜」大方向，具體要不要兩篇並列佐證，留待寫作定稿前依 T-GRAG 全文精讀結果決定；此處先誠實記錄兩版本引用不一致的既有落差。

**參考專案查證（2026-07-23 補齊，本節查得最強的參考專案）**：Zep 對應的開源實作 **Graphiti**（[getzep/graphiti](https://github.com/getzep/graphiti)，29,096★，2026-07-23 GitHub API 查證）——需明確區分兩件事：Zep **論文**本身未全文核實（上一段已誠實聲明），但 Graphiti **專案**是真實存在、規模龐大（在本論文查證過的所有參考專案中星數最高，甚至超過 RAGFlow）、持續維護的生產級開源程式碼，且其核心資料模型正是「邊上標記 `valid_at`／`invalid_at` 時間戳、舊事實不刪除而是標記失效」——與本節 3.5 節設計提案的雙時態邊機制高度吻合。這代表：即使 Zep 論文全文尚未核實，「雙時態邊在真實生產系統中可被實作、且已有大規模採用」這件事本身有強力的參考專案佐證，論文查證缺口與實作可行性佐證是兩條獨立的證據鏈，不應因為前者未完成就連帶低估後者的價值。T-GRAG 本身則查無對應的公開程式碼倉庫，是純學術原型；若 RQ5 納入正式貢獻，設計實作時建議同時參照 T-GRAG 的方法論文字與 Graphiti 的實際 schema 設計。

**與本論文的關係**：第三章 3.5 節目前提出的「舊邊標記 valid_to、新邊標記 valid_from」設計精神借鏡自 T-GRAG／Zep-Graphiti 的雙時態邊概念，但衝突偵測邏輯（同 subject+rel_type、不同 object 時觸發）與時態查詢過濾的具體實作為本論文自行設計，非直接沿用文獻演算法（見第三章 3.8 節誠實框架）。RQ5 與 RQ1（多跳推理）的交互影響（若保留歷史，圖中會同時存在多條時態邊，BFS 遍歷若不加時態過濾，路徑數會比現行基準線更多）已於第三章 3.5 節標示為實驗設計風險，本節不重複。

### 2.4.7 對應 3.6（自我精煉檢索迴圈，RQ3）

Lewis et al.（2020，🟢 NeurIPS 2020）提出的 RAG 架構是這條技術線的起點：以 dense retriever 檢索若干段落，seq2seq generator 在生成時對檢索結果做邊際化（marginalize），首次把「參數化記憶」與「非參數化檢索記憶」結合進同一個端到端訓練框架，直接回應第一章 1.1.1 節 LeCun（2022）指出的「持久性記憶缺失」問題。然而 RAG 的檢索是**單次、一次性**的——檢索發生在生成之前，生成過程中不會重新評估「目前的檢索結果是否足夠」，這個侷限催生了後續一系列「主動式／反思式檢索」的研究：

- **FLARE**（Jiang et al., 2023，🟢 EMNLP 2023）在生成過程中預測下一句話的內容，若預測句子中出現低置信 token，才觸發新一輪檢索，讓檢索時機由生成過程動態決定，而非固定在生成之前一次性完成。
- **Self-RAG**（Asai et al., 2023，🟢 ICLR 2024）訓練模型輸出「反思標記」（reflection tokens），讓模型自行判斷「是否需要檢索」「檢索結果是否支持回答」「回答品質如何」，把檢索決策內化為模型生成的一部分。
- **IRCoT**（Trivedi et al., 2022，🟢 ACL 2023）將檢索與思維鏈（Chain-of-Thought）推理交錯進行——每產生一步推理，就據此推理結果觸發下一輪檢索，特別針對多跳問答（multi-hop QA）任務設計。

**與本論文的關係**：這條文獻線的共同關注點是「檢索時機與方式」的動態化，本論文第三章 3.6 節的自我精煉迴圈（對應 RQ3）與此脈絡直接對話，但機制上有明確差異，不能混為一談——FLARE／Self-RAG／IRCoT 三者作用的對象都是**自由文字段落**的檢索與生成，觸發訊號來自生成模型自身的 token 置信度或反思標記；本論文的自我精煉迴圈作用於**結構化三元組 Context**，觸發訊號與圖遍歷本身的信心（如種子實體命中數、BFS 路徑長度）綁定，回補動作是「擴大 BFS 跳數」而非「重新檢索文字段落」。

**參考專案查證（2026-07-23 補齊）**：三篇皆有作者釋出的官方程式碼，可供對照演算法實作細節（而非僅依賴論文文字描述）：**Self-RAG**（[AkariAsai/self-rag](https://github.com/AkariAsai/self-rag)，2,410★）、**FLARE**（[jzbjyb/FLARE](https://github.com/jzbjyb/FLARE)，669★）、**IRCoT**（[StonyBrookNLP/ircot](https://github.com/StonyBrookNLP/ircot)，272★，2026-07-23 GitHub API 查證）。**誠實侷限**：三個官方倉庫本身**皆未實作圖結構檢索**（皆針對自由文字段落），與上一段已聲明的機制差異一致——這些倉庫的參考價值僅止於「反思式/主動式檢索觸發邏輯的具體實作範例」，不能作為本論文自我精煉迴圈（作用於三元組 Context）本身的直接程式碼佐證，此節仍應歸類為第三章 3.8 節「本論文自行設計的工程決策」，官方倉庫只是設計動機層級的參照。

### 2.4.8 對應 3.7（向量引導圖剪枝，RQ6，預留）

高連結度節點（Hub Node）在圖遍歷時容易造成路徑爆炸——BFS 從種子實體出發，一旦某一跳命中連結數極高的節點（如知識圖譜中常見的「總稱性」實體），下一跳待展開的候選集合會急遽膨脹。這個現象過去多僅見於工程實務的 naive 硬截斷處理，但近兩年已出現正式的學術命名與初步解法：

- **CatRAG**（Lau et al., 2026，🟡）《Breaking the Static Graph》明確將此問題命名為**「Static Graph Fallacy」**（靜態圖謬誤）——索引時固定的轉移機率忽略了查詢依賴的邊相關性，導致隨機遊走被引導至高連結度的「樞紐」節點，卻未能抵達與查詢真正相關的下游證據。CatRAG 建立在 HippoRAG 2 之上，提出查詢自適應的圖遍歷取代固定轉移機率。同一現象在多篇 2025-2026 論文中也被交叉引用為「Hub Bias」或「degree centrality bias」，確認這是被學界認可的現象，而非單一論文的孤立說法。
- **PathRAG**（Chen et al., 2025，🟡）提出流量式剪枝（flow-based pruning），處理冗餘的關聯路徑以改善檢索到的資訊連貫性，在 6 個資料集、5 個評估維度上優於基準方法。

**⚠️ 待補全文查證**：本節草稿原規劃引用 **He et al.（2024）G-Retriever**（採 PCST 子圖檢索處理大節點問題），但截至目前尚未下載全文、亦未在 `docs/論文/附錄與參考文獻.md` 的查核表中登錄核實，暫不作為確定引用，僅記錄為候選文獻，寫作定稿前需比照其他文獻的查核流程（arXiv／出版社頁面核實作者與出處）補齊。

**與本論文的關係**：第三章 3.7 節（向量引導圖剪枝，對應 RQ6，預留）的理論支撐正是 CatRAG 與 PathRAG 這兩篇文獻，但需誠實聲明：本論文目前規劃的設計是「查詢向量與出邊語意向量的相似度排序」，是一個**簡化的相似度剪枝方案**，並非 PCST（Prize-Collecting Steiner Tree）這類具備最佳化理論保證的嚴謹子圖檢索算法，也不是直接複用 CatRAG 或 PathRAG 的演算法實作。若 RQ6 最終納入正式貢獻，需要在第五章明確說明本論文簡化方案與文獻中嚴謹解法的差距，不能混淆兩者的方法論嚴謹程度。

**參考專案查證（2026-07-23 補齊）**：**PathRAG** 有官方程式碼（[BUPT-GAMMA/PathRAG](https://github.com/BUPT-GAMMA/PathRAG)，375★，2026-07-23 GitHub API 查證），可對照其流量式剪枝的實際實作方式。**CatRAG 查無可信的官方公開倉庫**（GitHub 搜尋結果皆為星數個位數、與論文機構/作者無關的同名專案，疑似巧合命名或習作），是 2026 年極新的論文，程式碼可能尚未釋出或以其他形式（如僅限審稿人存取）發布——這代表 RQ6 若採用 CatRAG 的查詢自適應遍歷精神，將**完全沒有可對照的參考程式碼**，只能依論文文字描述實作，屬於本節設計提案中風險最高的一塊，與第一章 1.2 節已將 RQ6 標為「風險最高」「暫緩」的判斷一致，此次查證進一步坐實了這個風險評估的依據（不只是「業界未解決」，也是「找不到任何可參考的既有實作」）。

## 2.5 評估方法論的橫向文獻回顧

GraphRAG 評估方法論本身在 2025-2026 年才開始被系統性檢視，而非早已成熟。本節內容不對應第三章單一節點，而是橫跨 RQ1、RQ3、RQ4a、RQ4b、RQ5、RQ6 共同的評估方法論基礎，故獨立成節，不併入 2.4 節的逐階段結構：

- **Han et al.（2025，同 2.4.3）**揭露的 LLM-as-Judge **位置偏誤**，是任何以 LLM 作為評審的實驗設計都必須警惕的方法論陷阱。
- **Zhou et al.（2025，🟡）**《In-depth Analysis of Graph-based RAG in a Unified Framework》指出過去的 Graph RAG 論文彼此未曾在同一基準上比較過（可重現性問題），建立統一開源基準以修正此問題。
- **GraphRAG-Bench**（Xiao et al., 2025，🟡，香港理工大學＋騰訊優圖）指出現有基準測試對長上下文、大規模異質語料的測試普遍不足，多停留在淺層多跳、簡答型、常識性問題，未能真正檢驗圖結構帶來的價值。
- **Dong et al.（2025，🟡）**《Knowledge-Graph Based RAG System Evaluation Framework》指出標準 RAGAS 類指標無法捕捉圖結構特有的性質（多跳結構、社群/叢集語意），提出結合語意社群分群的評分框架。
- **可追溯性評估**：本論文第一章 1.1.4／1.2 節採用的 AIS（Attributable to Identified Sources）框架，出自 **Rashkin et al.（2021/2023，🟢 期刊版 *Computational Linguistics* 49, 2023）**，量化生成內容是否可歸因於明確標識的來源，是第三章 RQ3／RQ4a／RQ4b 可追溯性次要指標的評估依據；**Shuster et al.（2021，🟢 EMNLP 2021 Findings）**的實證結果（檢索增強可將對話幻覺率降低 60% 以上）則是本論文「低幻覺率」產品目標的實證支撐。

**⚠️ 待補全文查證**：本節草稿原規劃引用 **Yang et al.（2018）HotpotQA**（多跳問答標準基準）與 **Es et al.（2023）Ragas**（RAG 評估框架命名依據），兩者截至目前皆尚未下載全文、亦未在附錄查核表中登錄核實，暫不作為確定引用。這兩篇文獻對第五章實驗設計（測試資料集與評估框架選型）有直接影響，需在著手設計第五章實驗前優先完成查證與下載。

**與本論文的關係**：本節梳理的方法論陷阱（位置偏誤、可重現性缺口、圖結構專屬性質未被標準 RAGAS 捕捉）直接構成第五章實驗設計的依據——至少需避免 Han et al. 指出的位置偏誤，且評估指標不能只套用一般 RAGAS，需針對圖結構檢索設計專屬評估項目。

### 2.6.1 文件解析模組（對應 3.1 總覽圖 `PARSE`，完整 Behavior Tree 見 `parser/README.md`）

**Whisper**（Radford et al., 2022/2023，🟢 ICML 2023）是本專案「音訊/影片語音轉譯」功能的核心依據；**Trafilatura**（Barbaresi, 2021，🟢 ACL-IJCNLP 2021 System Demonstrations）是本專案「網頁正文擷取」功能的核心依據。兩篇文獻的完整查證與 `parser/README.md`「學術文獻支撐」章節的對應關係見附錄與參考文獻.md。

### 2.6.2 向量化模組（對應 3.1 總覽圖 `VEC`，完整 Behavior Tree 見 `core/providers/embedding/README.md`）

向量化本身不是本論文的研究問題，完整文獻佐證與 Behavior Tree 收納於 `core/providers/embedding/README.md`，不寫入第三章正文，此處僅摘要其角色：**Sentence-BERT**（Reimers & Gurevych, 2019，🟢 EMNLP-IJCNLP 2019）證明句子向量間 cosine 相似度可作為有意義的語意距離度量，支撐本專案 chunk 向量平均值＋cosine 相似度的分類/分群/路由設計（與 2.4.1.1 的 Prototypical Networks 精神一致，兩者共同構成 `CMP` 節點的完整理論基礎）；**Dense X Retrieval**（Chen et al., 2023/2024，🟢 EMNLP 2024）與 **Rethinking Chunk Size**（Bhat et al., 2025，🟡）分別回應切塊粒度的選型依據；**MTEB**（Muennighoff et al., 2022/2023，🟢 EACL 2023）的核心發現「沒有單一模型在所有任務上都最佳」，是本專案 embedding provider 可替換設計的實證支撐；**ColBERT**（Khattab & Zaharia, 2020，🟢 SIGIR 2020）的多向量後期互動機制列為未來優化方向，非現行採用。

## 2.7 研究方法論誠實框架（呼應第三章 3.8）

本章各節引用的文獻，依第三章 3.8 節「研究方法論說明」的分類原則，同樣分兩類看待，不可混為一談：借用既有理論者（如 Schema.org 之於結構/語意驗證兩步機制、Self-RAG/FLARE/IRCoT 之於自我精煉迴圈、T-GRAG/Zep-Graphiti 之於事實時序保留策略），具體實作皆為本論文自行設計，非直接套用文獻演算法；本論文自行設計的工程決策者（雙層路由架構、實體別名消解機制、事實衝突偵測與時態查詢過濾、向量引導剪枝的相似度計算方式），目前業界與學界皆無現成解法可直接採用，這正是本論文宣稱研究貢獻之處。完整分類清單見第三章 3.8 節，本節不重複列舉，僅重申文獻探討章與系統設計章對同一組技術決策的分類判準必須一致。
