# 本地抽取與混合架構評估（技術決策記錄）

> **目的**：記錄關於個人知識管理與共享系統的「本地算力抽取」與「雲端/後端混合架構」討論。本文件將作為系統架構設計（對應論文第四章）與後續開發路線圖的工程決策依據。

---

## 一、 背景與核心痛點

在個人知識管理與共享的場景下，自動化知識圖譜建構（GraphRAG）面臨兩個工程瓶頸：
1. **極高的 Token/雲端算力成本**：若使用雲端大模型（如 GPT-4o、Claude 3.5）對大量筆記進行 SVO（主體-謂體-客體）關係抽取，API 費用將隨文件量呈指數級增長，開發者與使用者皆難以承受。
2. **抽取任務的耗時與中斷風險**：大批量文件的抽取任務通常需持續數小時，傳統的 Web 瀏覽器端執行極易因分頁關閉、系統休眠或記憶體不足（OOM）而中斷，且缺乏斷點續傳能力。

---

## 二、 核心決策一：本地資源抽取 (Edge Extraction)

為了解決 Token 費用並保護隱私，**「抽取任務」將完全移至使用者本地運行**，利用本地 CPU/GPU 算力進行。

### 1. 建議模型配置與分檔

針對不同使用者的硬體配置，軟體將提供「一鍵安裝/下載」的引導管理，並推薦以下三種模型檔次：

| 檔次 | 推薦模型/架構 | 模型大小 | 適用硬體 | 運作特點 |
| :--- | :--- | :--- | :--- | :--- |
| **1. 輕量極速檔** | `Qwen-2.5-1.5B-Instruct-GGUF` | ~1.2 GB | 無顯卡的文書機、舊電腦 | 純 CPU 運行，記憶體佔用極低 (僅需 4-8G RAM)，能完成基本 SVO 抽取。 |
| **2. 黃金平衡檔** | `Qwen-2.5-7B-Instruct-GGUF` (Q4_K_M) | ~4.7 GB | 主流筆電、Apple M 系列、有獨立顯卡 | 抽取精準度高，為目前開源最強尺寸，需 16G RAM。 |
| **3. 極致低配檔** | `GLiNER-medium` + `REBEL` | ~500 MB | 超老舊電腦 | 專用小模型（非 LLM），直接以 Python 在 CPU 運行，毫秒級完成抽取。 |

### 2. 模型分發策略
* **解耦軟體與模型**：軟體本體（Electron + Python）控制在 100MB~200MB。模型文件在使用者首次使用時，引導其「一鍵下載」或「對接本地已有的 Ollama（`localhost:11434`）」，避免安裝包過於臃腫。
* **模型存放路徑**：統一放在使用者目錄下（如 Windows: `C:\Users\<Username>\.world_graph_rag\models\`），避免系統權限與更新衝突。

## 三、 核心決策二：任務與功能解耦 (Hybrid Model Architecture)

將**「抽取任務（背景非同步）」**與**「問答任務（即時互動）」**所使用的模型完全解耦，以達到效能與成本的最大化。

### 1. 混合配置方案

```mermaid
graph TD
    subgraph Client [用戶本地端 (Edge)]
        Docs[原始文件 / Markdown] -->|1. 本地切片| Chunks[Chunks]
        Chunks -->|2. 本地抽取| LocalLLM[本地輕量模型 <br> Qwen-1.5B / REBEL]
        LocalLLM -->|3. 產生| SVO[SVO 三元組 & 向量]
    end

    subgraph Server [後端中樞 (Cloud/Server)]
        SVO -->|4. API 推送| Gateway[Ingestion Gateway]
        Gateway -->|5. 實體對齊與融合| Neo4j[(Neo4j 共享/個人圖譜)]
        UserQuery[使用者提問] -->|6. 混合檢索| Retrieval[Vector + GraphRAG]
        Retrieval -->|7. 檢索 Context| QA_LLM[問答大模型 <br> Gemini Flash / GPT-4o-mini / 本地 7B]
        QA_LLM -->|8. 回傳| Answer[生成回答 (帶 Citation)]
    end
```

1. **混合雲端方案（推薦）**：
   * **抽取**：在本地使用免費的 `Qwen-1.5B` 進行 background 抽取（完全免 Token 費）。
   * **問答**：使用雲端輕量大模型（如 `Gemini 1.5 Flash` 或 `GPT-4o-mini`，可由使用者填寫 API Key），確保問答時擁有極高的推理與語言整合能力，且因為問答頻次低，雲端 Token 費用極其低廉（每月通常小於 $0.1 USD）。
2. **純本地隱私方案**：
   * **抽取**：本地 CPU/GPU 跑小模型。
   * **問答**：提問時才將 `Qwen-7B` 或 `Llama-8B` 載入顯存，問完後釋放，避免持續佔用記憶體。

## 四、 核心決策三：背景守護行程與 SQLite 任務隊列

為了實現「數小時長任務」與「臨時中斷後開機接續」的能力，本機端不依賴網頁端運行，而是採用 **Daemon + Persistent Queue** 機制。

### 1. 守護行程 (Daemon) 實作
* 使用 **Electron 主進程** 作為背景 Daemon，常駐於系統托盤（System Tray），並支援隨系統啟動（開機自啟）。
* Electron 主進程在背景喚起本地的 Python 服務，進行任務管理與調度。

### 2. SQLite 任務狀態持久化
本地端建立 `task_queue.db` SQLite 資料庫，精確記錄每個檔案、每個 Chunk 的處理狀態：
* **狀態標記**：分為 `pending`, `processing`, `completed`, `failed` 與 `pending_upload`（抽取成功但尚未成功送達後端）。
* **斷點續傳**：當電腦開機或軟體重啟時，背景 Worker 掃描 SQLite，直接從上一次中斷的 `chunk_index` 開始繼續抽取，不重複消耗本地算力。

## 五、 後端作為「對接管道 (Gateway)」的職責

在此架構下，你的後端（雲端/伺服器）不再負擔高昂的 LLM 抽取算力，轉而專注於以下核心價值：

1. **統一對接管道 (Pipeline API)**：
   * 提供 `/api/ingest/graph-data` 接收本地端送來的結構化 SVO 與 Embedding。
   * 支持網絡中斷時的快取重送機制（由本地 SQLite 配合重試）。
2. **實體對齊與去重 (Entity Resolution)**：
   * 後端收到多個本地端送來的節點時，利用向量相似度與語法對齊（例如將「TSMC」、「台積電」合併為同一節點），維護全局圖譜的整潔。
3. **安全與權限隔離 (Namespace)**：
   * 在 Neo4j 中以標籤區分個人專屬圖譜與社群共享圖譜，確保個人隱私在外流的同時，也能實現知識共享。
