## 一、 技術導入清單與來源對照

| 導入技術 | 來源競品 | 解決本專案的什麼痛點 | 實作難易度 | 優先序 |
| :--- | :--- | :--- | :--- | :--- |
| **1. 雙層圖結構 (Lexical + Entity Graph)** | Neo4j LLM KG Builder | 回答無來源依據，缺乏「可追溯性/可解釋性」 | 中等 | 🔴 高 (核心) |

## 二、 具體技術導入規格與設計

### 1. 雙層圖結構 (Lexical + Entity Graph)
* **設計邏輯**：
  在 Neo4j 資料庫中，不要只儲存抽象的實體與關係。必須建立「文件層」與「圖譜層」的連結。
* **Neo4j Schema 調整**：
  * 新增 `:Chunk` 節點（屬性包含 `text`, `vector` 等）。
  * 原有實體 `:Entity` 節點。
  * 建立邊 `(Chunk)-[:HAS_ENTITY]->(Entity)`。
* **RAG 應用**：
  問答檢索時，透過 Cypher 語句從查到的實體反向尋找關聯的 `:Chunk`，並在 API 中將該 Chunk 的文字與 ID 作為 **Source Citation (資料來源標記)** 返回給前端。

### 2. 實體對齊與去重 (Entity Resolution)
* **設計邏輯**：
  LLM 抽取實體時，由於同義詞或縮寫，會產生重疊節點。
* **後端 Ingestion 處理**：
  在 `/api/ingest` 接收到本地上傳的實體列表時：
  1. **字串編輯距離篩選**：如果名稱高度相似（如「台積電」與「台積電公司」），直接合併。
  2. **向量語義篩選**：如果名稱不同但語義相似度極高（餘弦相似度 > 0.88），且屬於相同實體類型，則執行 Neo4j `MERGE`。
* **效果**：保持圖譜乾淨，顯著提高 GraphRAG 的檢索召回率（Recall）。

### 3. 雙軌非同步建圖 (Asynchronous Ingestion)
* **設計邏輯**：
  建圖需要數小時，但使用者需要「上傳即用」。
* **流程設計**：
  * **寫入第一軌 (即時)**：上傳文件 $\rightarrow$ 切片 (Chunking) $\rightarrow$ 向量化 (Embedding) $\rightarrow$ 寫入向量索引。此時系統立即回傳「上傳完成」，使用者已可進行基礎的 Vector RAG 問答。
  * **寫入第二軌 (背景非同步)**：任務寫入 SQLite 本地隊列 $\rightarrow$ 背景 Worker 逐一調用本地模型進行 SVO 關係抽取 $\rightarrow$ 推送三元組到後端寫入 Neo4j。

### 4. 本地資源抽取與功能解耦 (Edge Extraction & Model Decoupling)
* **設計邏輯**：
  抽取任務需要消耗大量算力，需將其完全本地化。
* **系統配置**：
  * **Extraction Model (本地)**：使用本地跑的 `Qwen-2.5-1.5B-GGUF` 或專用模型 `REBEL`。因為是背景非同步跑，速度慢一些也無妨，完全免費且不耗費雲端算力。
  * **QA/Generation Model (雲端/大模型)**：使用 `Gemini 1.5 Flash` 或 `GPT-4o-mini` API（或本地 7B 以上模型），只在使用者問答時觸發，反應速度快且推理能力強。

### 5. 雙時態有效期邊 (Bi-temporal Relation) — *進階優化*
* **設計邏輯**：
  解決個人知識管理中，資訊隨時間流逝而矛盾或失效的問題。
* **實作方式**：
  * Neo4j 中的所有關係邊（Relationships）增加屬性 `valid_from` (時間戳) 與 `valid_to` (預設為 `null`/無限)。
  * 當本地背景 Worker 抽取出新事實，且與舊事實產生邏輯衝突時（例如：舊關係為 `(A)-[職位:總經理]->(B)`，新關係為 `(A)-[職位:董事長]->(B)`）：
    * 不刪除舊邊，將舊邊的 `valid_to` 設定為目前時間（標記失效）。
    * 新增一條邊，其 `valid_from` 設定為目前時間。
  * 查詢時，Cypher 查詢預設只查詢 `valid_to IS NULL` 的 active 關係，但在歷史追溯時可以查詢任意時間點的圖譜狀態。

### 6. Namespace 輕量隔離與 Seed-Question 模式 (參考 WhyHow.ai)
* **設計邏輯**：
  提供輕量化的多知識庫隔離機制，並為特定專業領域提供動態 Schema 擴充能力。
* **實作方式**：
  * **Namespace 隔離**：不使用複雜的 RBAC 權限，在本地 SQLite、後端 API 以及 Neo4j 的節點/邊上，全面以 `namespace` (即 `kg_id`) 作為物理或邏輯隔離鍵。
  * **Seed-Question 模式 (選評)**：建圖前，允許用戶輸入代表性問題（如「我想問藥物與疾病的關聯」），LLM 分析問題自動生成 2~3 個專屬實體與關係 Schema，合併至系統固定的 30 種關係中進行抽取，提高特定領域的建圖精準度。
  * **架構決策 (Service-first)**：借鏡 WhyHow 棄用 SDK 的工程教訓，專案不開發為純 Python SDK，而是維持**獨立的 FastAPI 後端服務 (API-first)**，以利未來對接 Electron 或其他前端。

### 7. 本地輕量向量與段落共現分析 (參考 Obsidian 生態)
* **設計邏輯**：
  進一步降低本地資源消耗，並透過非大模型演算法（統計學方法）補足 SVO 抽取召回率不足的盲點。
* **實作方式**：
  * **本地輕量向量 (BGE-micro)**：本地背景 Worker 預設搭載 `BAAI/bge-micro-v2` 等極小嵌入模型，不需調用 API Key 與高記憶體顯卡，在純 CPU 下完成秒級 Chunk 向量化。
  * **段落共現分析 (Co-occurrence)**：在本地分塊與抽取時，對出現在同一個 Chunk 中的實體，不經大模型判斷，直接建立一條 `CO_OCCURS` 的關係邊。
* **效果**：大幅降低計算成本，並在 GraphRAG 檢索時能召回「雖無顯式主謂賓關係、但出現在同一個上下文中」的隱性關聯實體，大幅提高知識檢索的上下文廣度。

### 8. GraphRAG 混合問答檢索與語境生成 (Prompt Grounding)
* **設計邏輯**：
  如何真正將圖譜撈出來的事實關係（SVO）與向量 Chunk 結合，並轉化為 LLM 能夠理解的 Context 來生成對話回答。
* **實作方式**：
  1. **實體識別**：對使用者輸入的問答 Query，先提取核心實體（如 `台積電`、`NVIDIA`）。
  2. **圖檢索與路徑遍歷**：在 Neo4j 中執行 2-Hop 的 BFS 查詢，找出這兩個實體之間的所有連結路徑與相關三元組事實。
  3. **語法轉換**：將獲取的三元組 facts 自動轉譯為 XML 或 JSON 語意結構，寫入 System Prompt 中。
  4. **Grounding 生成**：LLM 讀取這些強制的「已知圖譜事實」來回答問題，並依據邊上的 `source_doc_id` 輸出對應的文檔來源標籤 (Citation)。

## 三、 進階優化與技術儲備 (未來工作 - 參考 Stardog)

為了避免引入重量級企業基礎設施 (Stardog 平台為閉源商業授權)，若後續要為系統加入「邏輯規則推導」以增加學術亮點，可導入並參考以下開源技術與套件：

### 1. 輕量級語意推理 (Semantic Rule Inference)
* **目的**：利用規則 (Rules) 自動由已抽取的顯式三元組推導出隱式關係，使 RAG 檢索更具備邏輯推理能力。
* **推薦 Python 開源套件**：
  * **[owlready2](https://github.com/pongac/owlready2)** (L-GPL 授權)：Python 最著名的本體 (Ontology) 推理與管理庫，可以直接在 Python 中載入 OWL 本體並調用 HermiT 推理器執行描述邏輯推理與一致性檢查。
  * **[rdflib](https://github.com/RDFLib/rdflib)** (BSD 授權)：Python 官方標準的 W3C 語意網開發套件，支持簡單的 RDFS 推理。
* **替代工程方案 (最易實作)**：
  在檢索層（Python 後處理或 Neo4j Cypher 層）直接手寫關聯規則（如 `A PREVENTS B 且 B CAUSES C ⟹ A PREVENTS C`），利用 **[NetworkX](https://github.com/networkx/networkx)** 進行快速圖遍歷與關係傳遞推導。
