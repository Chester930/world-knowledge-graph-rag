## 三、逐項痛點深入（Level 1 → Level 2）

### 痛點 1-2：建圖延遲與全局查詢（🔧 工程借鏡）

- **改善方向**：知識圖譜建構做成可選開關的步驟；文件上傳後第一軌立即向量化可用，第二軌背景非同步做 SVO 抽取；查詢分 Global（社群摘要）/ Local（BFS 鄰居擴散）兩種模式。
- **更細部改善項目**：
  - SQLite 任務隊列（`pending`/`processing`/`completed`/`failed`/`pending_upload`），支援斷點續傳
  - Electron 主行程作為背景 Daemon，常駐系統托盤
  - 社群偵測 + LLM 摘要寫入 `:Community` 節點（全局查詢用）——⚠️ 演算法選型見下方痛點 13，v1 用的 Louvain **不等同於** GraphRAG 的階層式 Leiden，不能視為「已經做了一部分」
- **完整規格**：`01_本地抽取與混合架構評估.md`、`02_競品借鏡與技術導入藍圖.md` §2-3

### 痛點 3：回答不可追溯（🔧 工程借鏡）

- **改善方向**：Neo4j 新增 `:Chunk` 節點與 `(Chunk)-[:HAS_ENTITY]->(Entity)` 邊。
- **更細部改善項目**：
  - 問答時反向查詢實體所屬 Chunk，回傳 `source_doc_id` 作為 Citation
  - 2-hop BFS 撈出的三元組轉譯為結構化 Context（XML/JSON）注入 System Prompt（GraphRAG Prompt Grounding，參考 Simple Graph Builder 概念延伸）
- **完整規格**：`02_競品借鏡與技術導入藍圖.md` §2-1、§2-8

### 痛點 4：實體去重（🔧 工程借鏡）

- **改善方向**：字串編輯距離篩選 → 向量相似度篩選（餘弦相似度 > 0.88 且同類型）→ Neo4j `MERGE`。
- **完整規格**：`02_競品借鏡與技術導入藍圖.md` §2-2

### 痛點 5：時序稽核（原標 🔧 工程借鏡，進階；⚠️ 2026-07-23 更正：已於 01 緒論升級為 🎓 研究問題型 RQ5，見上方「RQ 編號版本說明」）

- **改善方向**：關係邊新增 `valid_from`/`valid_to`，衝突時舊邊標記失效而非刪除。
- **狀態**：優先序中，非首波實作範圍（見 `ARCHITECTURE.md`）；2026-07-23 已於 `03_系統設計與方法論.md` § 3.5 補上正式 Behavior Tree（設計提案，尚未實作），核心文獻改採 T-GRAG（Li et al., 2025），本文件原引用的 Zep/Graphiti（Rasmussen et al., 2025）未下載全文，查證等級較低，完整說明見 `docs/論文/02_文獻探討.md` § 2.4.6。

### 痛點 6：雲端成本（🔧 工程借鏡）

- **改善方向**：抽取模型三檔次本地化（`Qwen-2.5-1.5B` / `Qwen-2.5-7B` / `GLiNER`+`REBEL`），問答模型走雲端輕量 API（`Gemini 1.5 Flash`/`GPT-4o-mini`）。
- **完整規格**：`01_本地抽取與混合架構評估.md`

### 痛點 7：抽取召回率盲點（🔧 工程借鏡，進階）

- **改善方向**：`BAAI/bge-micro-v2` 本地輕量向量化；同 Chunk 共現實體直接建 `CO_OCCURS` 邊。
- **狀態**：優先序中。

### 痛點 8：多知識庫路由（🎓 研究問題 RQ1）

- **現況**：WhyHow.ai 有 namespace 隔離機制存在，但**沒有任何競品驗證過**「輕量路由層 + BFS 知識層能否在準確率不顯著下降前提下改善延遲與可擴展性」這個效能取捨問題。
- **本論文的角色**：這是要親自做實驗回答的問題，不是抄一個現成架構就能解決。
- 對應：1.2 節 RQ1、`ARCHITECTURE.md`「命名空間隔離與服務形態」決策（namespace 隔離是**必要但不充分**的前置工程基礎，RQ1 驗證的是其上的路由效能，兩者不是同一件事）。

### 痛點 9：Hub Node 剪枝（🎓 研究問題 RQ4）

- **現況**：Neo4j Builder、RAGFlow 都沒有針對 Hub Node 路徑爆炸的專門機制；RAGFlow 的 Global/Local 模式是**繞開**問題（先摘要再查），不是**解決**問題（BFS 遍歷本身仍會爆炸）。
- **本論文的角色**：向量引導剪枝目前完全未實作，是四個 RQ 中風險最高的一項，去留已決定「暫緩，待 RQ1-3 進度明朗後再評估」（見 1.2 節）。

### 痛點 10-11：RQ2、RQ3（🎓 研究問題）

見 1.2 節「業界對照」欄，已有完整記錄，此處不重複。

### 痛點 12：實體/關係摘要整併（🔧 工程借鏡，2026-07-13 核對 Edge et al. 2024 官方文件時新發現）

- **GraphRAG 原文機制**：官方文件明確描述兩個獨立步驟——「Entity Summarization: LLM is prompted to combine the descriptions for every instance of an entity found across the text units into a single summary」，「Relationship Summarization」同理。也就是同一實體/關係若在多個文字片段被重複抽取出不同描述，GraphRAG 會用 LLM 把這些描述整併成單一摘要，才寫入圖譜。
- **本專案現況**：`svo_service.py` 目前的 SVO 抽取設計沒有這個整併步驟——三元組是逐段落直接產出並寫入，不同段落對同一實體的描述不會被合併摘要。這與痛點 4「實體去重（Entity Resolution）」不同：去重解決的是「同一實體被當成不同節點」，這裡要解決的是「同一實體/關係已經是同一節點，但其多筆描述沒有被整併成一份摘要」。
- **改善方向**：在實體去重（痛點 4）完成節點合併之後，新增一道 LLM 摘要步驟，把合併後節點的所有原始描述整併成單一摘要欄位。
- **狀態**：新發現，尚未評估優先序，待排入 `技術參考地圖.md`。

### 痛點 13：階層式 Leiden 社群偵測（🔧 工程借鏡，2026-07-13 核對 GraphRAG 演算法選型時新發現）

- **GraphRAG 原文機制**：查證確認 GraphRAG 使用 **Leiden 演算法**（Louvain 的改良版，解決 Louvain 可能產生內部不連通社群的問題），且是**遞迴、階層式**套用——對每個偵測出的社群持續遞迴切分子社群，直到達到葉節點社群（無法再切分），每一層都是對圖節點的互斥且完整覆蓋（mutually exclusive, collectively exhaustive）切分，才能支撐 Global Search 的分治式摘要。
- **本專案/v1 現況**：v1 的 `community_service.py` 用的是 **Louvain**（不是 Leiden），且**只做單層偵測**，只挑前 3 大社群做導讀/FAQ 生成（見智慧知識庫 v1 `04_對標NotebookLM...md` 方案七），不是遞迴階層式切分全圖。這跟 GraphRAG 的用法在演算法與使用目的上都不同，**不能視為已有部分實作**。
- **改善方向**：若要支撐痛點 2（全局查詢）與痛點 13 本身，需要換成 Leiden 演算法（`python-igraph`/`leidenalg` 套件）並改為遞迴階層式切分，而非沿用 v1 的單層 Louvain 邏輯。
- **狀態**：新發現，與痛點 2（全局查詢）高度相關，建議合併評估優先序。
