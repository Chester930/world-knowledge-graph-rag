## 一、雙層圖結構（Lexical Graph + Entity Graph）

- **來源**：Neo4j LLM Knowledge Graph Builder（[詳細文件](產品競品研究/02_Neo4j_LLM_Knowledge_Graph_Builder.md)）
- **技術描述**：知識圖譜中同時保留「文件/chunk 節點」（Lexical Graph）與「實體節點」（Entity Graph），兩者顯式建邊關聯，讓「回答可追溯到哪個原始 chunk」變成圖遍歷的一部分。
- **學術支撐**：Gutiérrez et al.（2024）*HippoRAG: Neurobiologically Inspired Long-Term Memory for Large Language Models*，NeurIPS 2024，arXiv:[2405.14831](https://arxiv.org/abs/2405.14831)。核心主張是複合知識圖譜同時整合「稀疏詞組層節點（實體/概念）」與「密集段落層節點（脈絡）」，透過雙層節點結構強化檢索的精確度與可追溯性，與 Neo4j Builder 的雙層圖設計是同一類架構思路。
- **參考開源實作**：
  - [neo4j-labs/llm-graph-builder](https://github.com/neo4j-labs/llm-graph-builder)（Apache-2.0，~4.7-4.9k★）
  - [OSU-NLP-Group/HippoRAG](https://github.com/OSU-NLP-Group/HippoRAG)（開源，NeurIPS'24 官方實作）
- **對本專案的適用性**：本專案目前的知識圖譜主要是實體圖層（SVO 三元組），沒有明確的「文件/chunk 節點」與實體圖分離並顯式關聯。這個缺口直接對應論文 1.1.4 節已主張的「可解釋知識問答」目標。
- **建議**：✅ **建議採用**。屬於架構層級的調整（需要在既有 schema 上新增 Document/Chunk 節點類型與對應邊），但學術與工業實作皆有先例可循，風險可控，投資報酬率高。

## 二、雙時態知識圖譜（Bi-temporal Knowledge Graph）

- **來源**：Zep (Graphiti)（[詳細文件](產品競品研究/03_Zep_Graphiti.md)）
- **技術描述**：每條圖邊記錄明確的有效期間（`t_valid`/`t_invalid`），新資訊使舊事實非破壞性失效（保留歷史，而非刪除或覆寫）。
- **學術支撐**：Rasmussen, Paliychuk, Beauvais, Ryan, Chalef（2025）*Zep: A Temporal Knowledge Graph Architecture for Agent Memory*，arXiv:[2501.13956](https://arxiv.org/abs/2501.13956)。
- **參考開源實作**：[getzep/graphiti](https://github.com/getzep/graphiti)（Apache-2.0，完全開源，是 Zep 現行開源策略核心）。⚠️ 注意 Zep Community Edition（完整平台）已停止維護，只有 Graphiti 引擎本身持續開源。
- **對本專案的適用性**：本專案目前僅有 `svo_service.py` 內建的連續性時序衰減（`_temporal_decay()`），是隱性權重淡出，沒有明確記錄「這條事實何時失效、被什麼取代」。Graphiti 的顯式時間戳更適合企業場景需要的「時間點查詢」與稽核追溯，且與已規劃的「精確 CRUD」目標互補（CRUD 處理使用者主動修正，雙時態機制處理系統自動偵測到矛盾時的處理方式）。
- **建議**：🟡 **建議評估後採用，非急迫**。實作複雜度高於第一項（需要修改邊的資料模型、寫入邏輯、查詢邏輯），建議在 RQ1-3 完成後、有餘力時再評估導入，或列為未來工作。

## 三、GraphRAG 選配化與 Global/Local 雙模式查詢

- **來源**：RAGFlow（[詳細文件](產品競品研究/05_RAGFlow.md)）
- **技術描述**：知識圖譜建構做成可選開關的步驟而非強制流程；查詢時區分「Global（社群摘要，回答全局性問題）」與「Local（鄰居擴散，回答局部細節問題）」兩種模式。
- **學術支撐**：Edge et al.（2024）*From Local to Global: A Graph RAG Approach to Query-Focused Summarization*（Microsoft GraphRAG），arXiv:[2404.16130](https://arxiv.org/abs/2404.16130)——**已在論文 1.1.3 節引用**，此技術點是同一篇文獻的工業落地案例，不需要新增引用，只需要在系統設計章節（第三、四章）標明是同一篇文獻的實作參考。
- **參考開源實作**：
  - [infiniflow/ragflow](https://github.com/infiniflow/ragflow)（Apache-2.0，84,718★，GitHub API 查證）
  - [microsoft/graphrag](https://github.com/microsoft/graphrag)（官方實作）
- **對本專案的適用性**：直接對應論文 1.1.3 節已識別的「建圖冷啟動成本過高」缺口（v1 對標文件的頭號弱點）——若知識圖譜建構本來就是選配步驟，就能對應到「向量先行、圖譜非同步建構」的雙軌設計。Global/Local 雙模式也直接補上本專案目前「僅有 Local（BFS 局部遍歷）、缺乏 Global（社群摘要全局查詢）」的缺口。
- **建議**：✅ **建議優先採用**。學術文獻已經在論文中引用、工業實作成熟度最高（84.7k★），且直接對應已識別的兩個缺口（冷啟動、全局查詢），是投資報酬率最高的一項。

## 四、實體去重（Entity Deduplication / Resolution）

- **來源**：RAGFlow（[詳細文件](產品競品研究/05_RAGFlow.md)）
- **技術描述**：對抽取出的實體做去重/消歧，避免同一實體因不同名稱寫法（如「勞基法」vs「勞動基準法」）被當成不同節點。
- **學術支撐**：這是一個廣泛的經典研究領域（record linkage / entity resolution），近期 LLM 時代的整理性文獻：*LLM-empowered Knowledge Graph Construction: A Survey*，arXiv:[2510.20345](https://arxiv.org/abs/2510.20345)，將實體去重列為知識圖譜自動化建構的關鍵挑戰之一。沒有單一「奠基性」論文可引用，建議寫作時以此類近期綜述作為背景文獻，而非尋找單一開創性論文。
- **參考開源實作**：RAGFlow 本身（同上）；此外 `ConceptRepository` 若已有相關實體對齊邏輯，可視為本專案既有的差異化優勢（見 Neo4j Builder 文件第五節已指出對標產品普遍缺乏此機制）。
- **對本專案的適用性**：低成本、高投資報酬率的優化點，若本專案 SVO 抽取管線尚無此機制，應優先評估。
- **建議**：✅ **建議採用**，且是本清單中實作成本最低的一項，建議近期排入。

## 五、Seed-Question 模式（種子問題驅動抽取）

- **來源**：WhyHow.ai（[詳細文件](產品競品研究/07_WhyHow_ai.md)）
- **技術描述**：讓使用者用自然語言描述想問的問題，系統據此動態調整本體/抽取重點，作為固定 schema 抽取之外的輔助模式。
- **學術支撐**：查證過程**未找到直接對應的學術文獻**——這是產業界（WhyHow.ai）提出的產品設計模式，未見學術論文正式研究此機制的效果。若要引用，只能以產品案例引用，不能宣稱有學術理論支撐。
- **參考開源實作**：[whyhow-ai/knowledge-graph-studio](https://github.com/whyhow-ai/knowledge-graph-studio)（MIT，929★/107 forks，主力產品；舊版 SDK `whyhow-ai/whyhow` 已棄用）。
- **對本專案的適用性**：本專案目前只有一種抽取模式（固定 30 種關係類型），這個模式可作為封閉式抽取之外的**互補模式**（非取代），對「使用者不確定自己需要哪些關係類型」的冷啟動場景較友善。
- **建議**：🔵 **觀察，非急迫**。缺乏學術文獻支撐，且與本專案封閉式抽取的核心研究設計（RQ2）方向不同，建議列為長期觀察項目，不排入近期實作。

## 六、開放式關係抽取（Open Relation Extraction）

- **來源**：Obsidian + Simple Graph Builder（[詳細文件](產品競品研究/08_Obsidian生態.md)）
- **技術描述**：不限定關係詞彙表，讓 LLM 自由產生關係動詞，而非套用固定的封閉關係類型清單。
- **學術支撐**：Banko, Cafarella, Soderland, Broadhead, Etzioni（2007）*Open Information Extraction from the Web*，IJCAI 2007——Open Information Extraction（OpenIE）領域的開創性論文，提出 TextRunner 作為首個此類系統。
- **參考開源實作**：[junhewk/simple-graph-builder](https://github.com/junhewk/simple-graph-builder)（MIT，Obsidian 外掛，10 種固定實體類型但關係為自由動詞）。
- **對本專案的適用性**：**這一項不是「要不要採用」的問題**——本專案的 RQ2 正是要驗證「封閉式 30 種關係類型 vs 開放式關係抽取」的 trade-off，Simple Graph Builder 與 OpenIE 文獻剛好是開放式路線的具體業界與學術對照，應該保留在論文的比較討論裡，而不是拿來替換本專案現有設計。
- **建議**：⚪ **不適用「採用」框架，屬於研究對照組**。已於 1.2 節 RQ2 業界對照欄位中記錄，不需要額外的技術導入決策。

## 七、本地輕量 Embedding（建圖冷啟動優化）

- **來源**：Obsidian + Smart Connections（[詳細文件](產品競品研究/08_Obsidian生態.md)）
- **技術描述**：使用輕量本地嵌入模型做語意搜尋，不需要依賴外部 API 即可達到基本可用，降低冷啟動延遲與成本。
- **學術支撐**：Xiao, Liu, Zhang, Muennighoff（2023）*C-Pack: Packaged Resources To Advance General Chinese Embedding*（BGE / BAAI General Embedding），arXiv:[2309.07597](https://arxiv.org/abs/2309.07597)。
- **參考開源實作**：[FlagOpen/FlagEmbedding](https://github.com/FlagOpen/FlagEmbedding)（BGE 官方倉庫，開源）；Smart Connections 外掛本身。
- **對本專案的適用性**：直接對應 v1 對標文件點名的頭號弱點「秒級可用」——本地輕量嵌入模型是可以立即嘗試的技術方向，若本專案已使用 `core/providers/embedding/local.py`，可評估是否已採用同類輕量模型，或需要切換。
- **建議**：🟡 **建議評估現況後決定**。需要先確認本專案現有本地 embedding provider 的模型選擇與效能，若尚未使用同級輕量模型，建議優先評估切換。

## 八、段落共現分析（Co-occurrence Analysis）

- **來源**：Obsidian + InfraNodus（[詳細文件](產品競品研究/08_Obsidian生態.md)）
- **技術描述**：分析文本中詞彙的共現關係，補足純結構化三元組抽取可能遺漏的「同段落內語意相關但沒有明確 SVO 關係」的資訊。
- **學術支撐**：Paranyushkin（2019）*InfraNodus: Generating Insight Using Text Network Analysis*，The Web Conference (WWW'19)，DOI: [10.1145/3308558.3314123](https://dl.acm.org/doi/10.1145/3308558.3314123)，同儕審查會議論文。
- **參考開源實作**：InfraNodus 本身是商業產品（[infranodus.com](https://infranodus.com/)），**核心分析演算法未見開源實作**，若要導入需自行依論文方法重新實作，不能直接參考程式碼。
- **對本專案的適用性**：可作為 SVO 抽取的補充訊號構想，用於彌補顯式關係抽取的召回率盲點。
- **建議**：🔵 **觀察，非優先**。學術文獻存在但工程參考成本較高（無開源實作可抄），建議列為長期研究方向而非近期工程項目。
